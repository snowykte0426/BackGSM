## 분산 환경에서 Lock이 필요한 이유

여러 인스턴스가 동시에 같은 자원을 수정하면 중복 처리, 재고 오차, 중복 결제 같은 문제가 생길 수 있습니다.

## Lock 선택 기준

### 비관적 Lock
- DB 레벨에서 직접 잠금
- 같은 데이터 행에 대한 강한 정합성이 필요할 때 적합
- 트랜잭션 범위가 길면 병목이 생길 수 있음

### Redis 기반 Lock
- 여러 애플리케이션 인스턴스 간 분산 제어
- 빠르고 유연하지만, 만료 시간과 장애 상황을 신중히 다뤄야 함

### Redis Pub/Sub Lock을 고려하는 상황
- 이벤트 기반 알림이 필요할 때
- 대기 중인 워커에게 빠르게 해제 이벤트를 전달하고 싶을 때

## 어떤 기준으로 고르나?

- 강한 정합성과 단일 DB 레코드 보호: 비관적 Lock
- 여러 인스턴스 간 작업 중복 방지: Redis 분산 Lock
- 둘을 같이 쓰는 경우도 있음

## 분산 트랜잭션과 SAGA

MSA에서는 하나의 로컬 DB 트랜잭션으로 전체 정합성을 묶기 어렵습니다.
그래서 보통 **로컬 트랜잭션 + 이벤트 + 보상 트랜잭션** 조합인 SAGA를 사용합니다.

## 왜 2PC보다 SAGA를 쓰나?

### 2PC
- 강한 일관성
- 참여자 전원이 조율에 참여
- 지연과 결합도가 큼

### SAGA
- 서비스 자율성이 높음
- 장시간 분산 워크플로우에 적합
- 대신 일시적 불일치와 보상 로직을 감수해야 함

## 면접 포인트

- Lock과 트랜잭션 전략은 같은 문제가 아닙니다.
- "무결성 보장 방식"을 DB 잠금, 분산 잠금, 이벤트 기반 보상으로 나눠 설명하면 좋습니다.
