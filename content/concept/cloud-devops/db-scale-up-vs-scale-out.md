## 스케일 업

하나의 DB 서버의 CPU, 메모리, 디스크 성능을 높이는 방식입니다.

### 장점
- 구조가 단순함
- 애플리케이션 수정이 적음
- 트랜잭션과 일관성 관리가 상대적으로 쉬움

### 단점
- 물리적 한계가 있음
- 장비 비용이 급격히 증가할 수 있음
- 단일 장애 지점 문제가 남아 있음

## 스케일 아웃

DB 노드 수를 늘려 부하를 분산하는 방식입니다.

### 장점
- 수평 확장이 가능함
- 읽기 부하 분산에 유리함
- 고가용성 구성을 만들기 좋음

### 단점
- 운영 복잡도가 증가함
- 데이터 분산, 일관성, 트랜잭션 처리 전략이 필요함
- 단순히 서버 수만 늘린다고 쓰기 성능이 자동 향상되지는 않음

## 왜 스케일 업만 하지 않을까?

스케일 업은 직관적이지만, 어느 순간 더 이상 장비를 키우기 어렵거나 비용 대비 효율이 나빠집니다.
또한 장애가 나면 큰 단일 노드 하나가 통째로 영향을 받습니다.

## 스케일 아웃에서 샤딩만 답일까?

그렇지는 않습니다. 실제로는 여러 전략이 있습니다.

### Read Replica
- 읽기 요청을 복제본으로 분산
- 읽기 많은 서비스에서 효과적

### Caching
- Redis 같은 캐시로 DB 조회 자체를 줄임
- 가장 빠르게 체감 성능을 올릴 수 있는 경우가 많음

### CQRS / 비동기 처리
- 읽기 모델과 쓰기 모델을 분리
- 검색, 통계, 집계성 조회를 별도 저장소로 넘길 수 있음

### 파티셔닝
- 하나의 DB 안에서 테이블을 구간별로 나눔
- 샤딩보다 단순하지만 데이터 관리에 도움됨

## 면접 포인트

- 스케일 업은 단순성, 스케일 아웃은 확장성과 가용성이 핵심입니다.
- 읽기 확장과 쓰기 확장은 난이도가 다릅니다.
- "샤딩"만 말하지 말고 복제, 캐시, 파티셔닝도 함께 언급하면 좋습니다.
