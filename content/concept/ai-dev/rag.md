## RAG란?

LLM의 **한계(최신 정보 부재, 할루시네이션)**를 극복하기 위해, 외부 지식 베이스에서 관련 정보를 검색해 LLM 응답 생성에 주입하는 아키텍처입니다.

```
일반 LLM:  질문 → LLM → 답변 (학습 데이터만 사용)
RAG:        질문 → 검색 → 관련 문서 → LLM → 근거 있는 답변
```

## RAG 파이프라인

### 1. 인덱싱 (오프라인)

```
문서 → 청킹(Chunking) → 임베딩(Embedding) → 벡터 DB 저장
```

### 2. 검색 및 생성 (온라인)

```
사용자 질문 → 질문 임베딩 → 유사도 검색(Top-K)
→ 관련 문서 청크 → 프롬프트에 주입 → LLM → 최종 답변
```

## 임베딩(Embedding)

텍스트를 **고차원 벡터(숫자 배열)**로 변환합니다. 의미가 비슷한 문장은 벡터 공간에서 가까운 위치에 놓입니다.

```
"서울의 날씨" → [0.12, -0.34, 0.78, ...]  (1536차원 등)
"서울 기온"   → [0.13, -0.31, 0.75, ...]  ← 유사!
```

## 주요 벡터 DB

| DB | 특징 |
|----|------|
| **Pinecone** | 완전 관리형, 가장 대중적 |
| **Chroma** | 로컬/경량, 개발·프로토타입용 |
| **Weaviate** | 오픈소스, 멀티 모달 지원 |
| **pgvector** | PostgreSQL 확장, 기존 DB 활용 |
| **Qdrant** | Rust 기반, 고성능 |

## 청킹 전략

| 전략 | 설명 |
|------|------|
| Fixed-size | 고정 크기로 분할 (간단) |
| Recursive | 문단·문장 경계 우선 분할 |
| Semantic | 의미 단위로 분할 (고품질) |
| Document-level | 문서 구조(제목/섹션) 기반 |

## RAG vs Fine-tuning

| 구분 | RAG | Fine-tuning |
|------|-----|-------------|
| 지식 업데이트 | 실시간 가능 | 재학습 필요 |
| 비용 | 검색 비용 | GPU 학습 비용 |
| 근거 제시 | 출처 표시 가능 | 어려움 |
| 적합 상황 | 최신·사내 문서 Q&A | 스타일·도메인 특화 |

## 고급 RAG 기법

- **HyDE**: 가상 답변을 먼저 생성 후 검색에 활용
- **Re-ranking**: 검색 결과를 Cross-encoder로 재정렬
- **Query Expansion**: 질문을 여러 형태로 변환해 검색
- **GraphRAG**: 지식 그래프 기반 검색 (Microsoft)
