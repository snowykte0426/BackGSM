## 무조건 추상화가 많다고 좋은가?

아닙니다.
SOLID를 과하게 적용하면 오히려 클래스 수가 불필요하게 늘고 구조가 복잡해질 수 있습니다.

## 현실적인 기준

- 변경 가능성이 큰가?
- 구현체 교체 가능성이 있는가?
- 테스트 분리가 중요한가?
- 팀이 이해 가능한 복잡도인가?

## 좋은 적용 예

- 외부 연동 포트 분리
- 전략 패턴으로 조건 분기 제거
- 큰 서비스의 책임 분리

## 과한 적용 예

- 구현이 하나뿐인데 의미 없는 인터페이스 남발
- 단순 로직에도 과도한 패턴 도입

## 면접 포인트

- SOLID는 교과서 원칙이지만, 실무에선 균형 있게 적용해야 한다고 말하면 좋습니다.
