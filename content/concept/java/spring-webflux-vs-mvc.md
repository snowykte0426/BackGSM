## Spring MVC

- 전통적인 Servlet 기반
- 요청당 스레드 할당 모델
- 동기식 코드 작성이 직관적
- 일반적인 CRUD 백엔드에서 가장 널리 사용

## Spring WebFlux

- Reactive Streams 기반
- 논블로킹 I/O 지향
- `Mono`, `Flux` 중심 프로그래밍 모델
- 높은 동시성과 스트리밍 처리에 적합

## 차이의 핵심

### MVC
- 이해와 디버깅이 쉬움
- 생태계와 팀 숙련도가 높음
- 블로킹 드라이버와 잘 맞음

### WebFlux
- I/O 대기 시간이 긴 작업에서 효율적일 수 있음
- 적은 스레드로 많은 연결을 처리하기 좋음
- 하지만 코드 복잡도와 학습 비용이 큼

## 왜 항상 WebFlux를 쓰지 않을까?

- DB 드라이버, 외부 API, 팀 경험이 모두 논블로킹 친화적이어야 효과가 큼
- 한 부분이라도 블로킹이면 기대한 효과가 줄어듦
- 복잡성을 감수할 만큼의 트래픽/요구사항이 아니면 MVC가 더 실용적일 수 있음

## 면접 포인트

- WebFlux는 "무조건 빠르다"가 아니라 적합한 문제에서 장점이 있다는 식으로 설명해야 합니다.
- 생산성, 디버깅, 팀 숙련도까지 같이 이야기하면 더 좋습니다.
