## 왜 도입하는가?

비즈니스 로직이 프레임워크, DB, 외부 API 구현체에 강하게 묶이면 테스트와 변경이 어려워집니다.
헥사고날 아키텍처는 핵심 도메인을 바깥 의존성으로부터 분리하려는 접근입니다.

## 핵심 구성

### Port
- 애플리케이션이 바깥과 상호작용하는 인터페이스
- 입력 포트(use case), 출력 포트(repository, external client)로 생각할 수 있음

### Adapter
- Port를 실제 기술로 구현한 계층
- 예: REST Controller, JPA Repository Adapter, Kafka Adapter

## 장점

- 비즈니스 로직 테스트가 쉬움
- 외부 기술 교체 영향이 줄어듦
- 유스케이스 중심 구조를 만들기 좋음

## 단점

- 파일 수와 추상화 계층이 늘어남
- 소규모 프로젝트에서는 과할 수 있음
- 팀 학습 비용이 있음

## 학습곡선은 어떻게 다루나?

- 모든 팀원이 처음부터 완벽히 이해하길 기대하기보다
- 템플릿, 예제 모듈, 코드리뷰 기준, 패키지 규칙을 제공하는 방식이 현실적입니다.

## 면접 포인트

- "좋아 보여서"가 아니라 테스트성, 변경 용이성, 의존성 역전 필요 때문에 도입했다고 설명하는 것이 좋습니다.
- 장점만 말하지 말고 과도한 추상화 문제도 함께 인정하는 것이 낫습니다.
